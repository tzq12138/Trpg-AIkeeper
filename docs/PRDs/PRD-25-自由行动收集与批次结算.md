# PRD-25 - 自由行动收集与批次结算

**版本**: 1.0  
**状态**: 待开发  
**定位**: 多玩家自由发言到 AI KP 回合输入的桥梁

## 1. 背景

玩家线下自由讨论并随时用手机提交行动。如果每条行动立刻触发 AI，剧情会被打碎；如果完全等待房主，人类 KP 又回来了。系统需要行动收集窗口和批次结算。

## 2. 目标

- 收集一段时间内的玩家行动。
- 将行动归并为 `ActionBatch`。
- 让玩家看到自己的行动已进入本轮。
- 将批次交给 AI KP 主持循环。

## 3. 范围边界

**包含**: 行动窗口、批次状态、冲突处理、批次提交。  
**不包含**: 严格回合制、队长投票、复杂战斗先攻。

## 4. 用户故事

| ID | 用户故事 | 优先级 |
|---|---|---|
| US-25-1 | 作为玩家，我可以自然地随时提交行动。 | P0 |
| US-25-2 | 作为 AI KP，我需要一组行动作为当前回合输入。 | P0 |
| US-25-3 | 作为玩家，我想知道自己是否被纳入本轮。 | P0 |

## 5. 功能需求

1. Engine 在场景中维护行动收集窗口。
2. 新行动进入 `queued`。
3. 到达时间、人数或 AI 判断条件后生成 `ActionBatch`。
4. 批次包含玩家行动、时间、角色、声明意图和转写文本。
5. 冲突行动交给 AI KP 判断顺序或补问。
6. 批次状态下发给相关玩家。
7. 批次结算完成后所有 action completed。

### 5.1 行动状态映射

| 服务端事件 | 服务端状态 | 前端 `actionState` | UI 表现 |
|---|---|---|---|
| HTTP 202 | `queued` | `SUBMITTING` | 输入按钮置灰，显示提交中 |
| `s2c_action_queued` | `queued` | `SUBMITTING` | 显示“已收到，等待归并” |
| `s2c_action_batched` | `batched` | `RESOLVING` | 显示“守秘人裁决中” |
| `s2c_action_completed` | `completed/rejected/timeout` | `IDLE` | 输入恢复，显示结算结果或失败原因 |

## 6. 接口/事件依赖

| 类型 | 名称 | 用途 |
|---|---|---|
| Type | `ActionBatch` | 行动批次 |
| Event | `s2c_action_queued` | 已接收 |
| Event | `s2c_action_batched` | 已归并 |
| Event | `s2c_action_completed` | 已结算 |
| Intent | `voice_command/dialogue/skill_check` | 行动来源 |

## 7. 状态与错误处理

- 场景锁定时新行动进入下一批或返回等待。
- 明显重复行动合并并提示玩家。
- 无效行动 rejected 并说明原因。
- 批次超时未处理时触发 AI KP 重试。

## 8. 验收标准

- 多名玩家可在同一窗口提交行动。
- 玩家能看到 queued 和 batched 状态。
- AI KP 接收的是批次而不是单条散乱输入。
- 批次完成后解锁对应玩家。

## 9. 测试场景

1. 三名玩家 10 秒内提交行动，生成同一批次。
2. 玩家重复提交相同行动，系统合并。
3. 场景锁定时行动进入下一批。
4. 批次结算后所有 action completed。

## 10. 风险依赖

- 窗口过短会打碎叙事，过长会让玩家等待。
- 战斗场景后续可能需要严格回合制扩展。
- 依赖 PRD-15 的玩家回执链。
