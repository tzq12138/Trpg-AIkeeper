# PRD-15 - 玩家行动回执与 AI 归并反馈

**版本**: 1.0  
**状态**: 待开发  
**定位**: 玩家提交行动后的可见状态链

## 1. 背景

自由发言 + AI 归并模式下，玩家行动不一定立刻结算。如果手机只显示“已发送”，玩家会不知道 AI KP 是否听懂、是否纳入本轮、是否已经结算。需要一条清晰的行动回执链。

## 2. 目标

- 展示录音中、转写中、已提交、等待归并、已归并、结算中、已结算。
- 让玩家知道自己的行动是否进入当前批次。
- 支持 AI KP 补问或拒绝无效行动。
- 为纠错请求提供最近行动上下文。

## 3. 范围边界

**包含**: `ActionReceipt`、`ActionBatchStatus`、回执 UI、补问提示。  
**不包含**: 任意撤回行动、玩家投票排序、复杂队伍聊天。

## 4. 用户故事

| ID | 用户故事 | 优先级 |
|---|---|---|
| US-15-1 | 作为玩家，我想知道我刚说的话有没有被系统听懂。 | P0 |
| US-15-2 | 作为玩家，我想知道我的行动有没有进入本轮结算。 | P0 |
| US-15-3 | 作为 AI KP，我需要对含糊行动进行补问。 | P1 |

## 5. 功能需求

1. 每个提交动作生成本地 `ActionReceipt`。
2. STT 返回后展示转写文本，允许玩家在短时间内取消提交或确认提交。
3. Engine 接收后回执状态为 `queued`。
4. 行动进入批次后状态为 `batched`，显示批次编号或“本轮行动”。
5. AI KP 开始结算后状态为 `resolving`。
6. 收到 `s2c_action_completed` 后状态为 `resolved/rejected/timeout`。
7. AI KP 需要补问时下发 `s2c_clarification_prompt`。

## 6. 接口/事件依赖

| 类型 | 名称 | 用途 |
|---|---|---|
| Type | `ActionReceipt` | 单个玩家行动状态 |
| Type | `ActionBatchStatus` | 行动批次状态 |
| Event | `s2c_action_queued` | Engine 已接收 |
| Event | `s2c_action_batched` | 已纳入批次 |
| Event | `s2c_clarification_prompt` | AI KP 补问 |
| Event | `s2c_action_completed` | 结算结束 |

## 7. 状态与错误处理

- STT 文本为空时不创建 queued 回执。
- 发送失败时回执显示失败并允许重试。
- 批次超时未结算时提示“AI KP 正在整理行动”。
- rejected 必须带玩家可理解的原因。

## 8. 验收标准

- 玩家提交行动后能看到完整状态变化。
- 已归并和已结算状态可区分。
- 补问不会直接改状态，需要玩家再次提交。
- 最近行动可被 PRD-18 用于澄清。

## 9. 测试场景

1. 语音提交后从 transcribing 到 queued。
2. Engine 将多个玩家行动归入同一 batch。
3. AI KP 对含糊行动发起补问。
4. completed 到达后回执显示结果摘要。

## 10. 风险依赖

- 依赖 PRD-25 的批次结算事件。
- 过多状态可能让玩家困惑，需要文案简短。
- 本地确认窗口过长会拖慢游戏节奏。
