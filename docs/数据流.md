[玩家意图提交] ↓ (阶段一：Python 资产校验层) -> 物理防线，验证拥有权与目标存在 ↓ (阶段二：LLM 机制编译器) -> 将发散交互“编译”为标准规则书检定 JSON ↓ (阶段三：Python 规则引擎绝对裁决) -> 数学发牌，执行内核暗骰与资产真实落库 ↓ (阶段四：LLM 叙事渲染器) -> 结果包装，生成最终多端演出与战术下发

````

#### 阶段一：Python 资产校验层（物理防线）
* **职责**：不消耗 Token，由 Python 代码进行死板的物理和拥有权对账，阻断“凭空造物”。
* **校验内容**：
  1. **持有校验**：查询当前角色的 `inventory`，确认该物品逻辑 ID 确实存在。
  2. **目标校验**：若意图涉及场景内其他实体（如 NPC、怪物），确认该实体在当前 `sceneId` 内活跃。
* **异常处理**：校验失败则直接返回 HTTP 400/403 驳回，UI 解锁。

#### 阶段二：LLM 机制编译器（规则投射）
* **职责**：调用极轻量、高响应的模型（如 GPT-4o-mini），利用 Function Calling（函数调用）能力将“物理描述”翻译为“规则书标准检定代码”。**此阶段严禁大模型生成任何小说文本。**
* **输入上下文**：玩家当前动作描述 + 物品材质属性标签 + 目标当前状态。
* **编译器输出 Schema (JSON)**：
```json
{
  "triggeredMechanic": "skill_check" | "auto_success" | "auto_failure",
  "skillName": "斗殴",
  "difficulty": "hard",
  "itemConsumed": true,
  "consequence": {
    "success": { "damage": "1d3", "effect": "窒息" },
    "failure": { "damage": "0", "effect": "激怒目标" }
  }
}
````

#### 阶段三：Python 规则引擎绝对裁决（数学执行）

- **职责**：依据阶段二编译出的标准指令，由 Python 内核执行纯代码硬核结算，严禁大模型篡改数学事实。
    
- **执行步骤**：
    
    1. 读取玩家面板对应的 `skillName` 数值，计算当前 `difficulty` 下的成功门限。
        
    2. 服务器内存生成 `1-100` 权威随机数（暗骰）。
        
    3. 根据 `itemConsumed: true`，执行原子操作，将该物品从 Inventory 移出。
        
    4. 计算最终属性扣减（如 HP/SAN 变更），递增 `stateVersion`。
        

#### 阶段四：LLM 叙事渲染器（结果包装）

- **职责**：将阶段三冷冰冰的数学事实（如：大失败、物品销毁、伤害0）组装为契合剧本氛围的最终演出文本。
    
- **输出下发**：
    
    - 向 **Host 端（大屏）** 投影：封装为 `s2c_reveal_transaction` 事务包，严格按序执行演出（3D 骰子滚出 `85` -> 血条闪红 -> 字幕逐字打印出卡壳和断裂描述）。
        
    - 向 **Player 端（手机）** 投影：下发 `s2c_state_patch`（扣除背包手套）和 `s2c_tactical_prompt`（解禁下一步战术按钮）。
        

## 📱 第二部分：后验叙事合理性主张机制（Retroactive Claim）

### 1. 核心定义

当玩家试图使用一个当前**不在角色卡背包（Inventory）内**的物品，但基于其**职业背景、人设、常识**进行合理解释（如：“我是医生，口袋里常备几双无菌塑胶手套极其合理”）并当场掏出的行为，系统通过「背景追溯导入机制」提供支持。

### 2. 三阶段对账与合流控制链

#### 步骤一：意图拦截与类型降级 (REST 网关)

当玩家通过自然语言发起 claim 意图时，由于资产库检索不到该物品，系统将其 `intentType` 动态重标记为 `retroactive_item_claim`，并将玩家主张的借口文本封装进 `params`。

#### 步骤二：人设-物品标签矩阵对账 (Python Engine)

Python 内核启动纯代码的“职业契合度矩阵”计算，通过莱文斯坦编辑距离和标签交集进行判定，划分出三条严格的硬编码分支：

1. **完胜分支（Auto-Pass）**：
    
    - _判定条件_：角色职业标签（如 `医疗`）与主张物品叙事标签（如 `防护`、`医疗`）高度交织，且该物品属于常识级物品（`baselineAccess: "common_professional"`）。
        
    - _内核操作_：Python 引擎认为完全合理，在内存中直接触发标准的 RFC 6902 `s2c_state_patch` 属性变更，将物品强行强制插入（add）到该玩家背包，标记 `source: "backstory"`，随后合流进入标准物品使用流水线。
        
2. **争议分支（Roll Required）**：
    
    - _判定条件_：职业与物品存在关联但属于管制/危险品（如：医生尝试从口袋掏出一把锋利的手术刀）。
        
    - _内核操作_：规则引擎强制打断，要求触发一次「幸运（Luck）」**或**「知识（Know）」暗骰。判定通过则允许导入物品并扣除幸运；判定失败则直接驳回。
        
3. **违和分支（Refused）**：
    
    - _判定条件_：职业背景与物品跨度过于荒谬（如：流浪汉尝试从口袋里摸出一支无菌注射器或对讲机）。
        
    - _内核操作_：引擎直接熔断，返回 HTTP 409/400 冲突错误，给玩家推送“主张失败”的 Toast 提示，不产生任何 AI 消耗。
        

#### 步骤三：双端多模态分流投影

对账通过后，该动作产生的事务进行跨端可见性切割：

- **Host大屏投影**：`s2c_reveal_transaction` 仅包含纯文本剧情 step：“约翰医生冷静地从白大褂口袋里摸出一副常备的无菌塑胶手套戴上……”。
    
- **当事玩家手机**：收到 `s2c_state_patch` 增量补丁，背包格瞬间刷新出该手套，同时通过 `s2c_action_completed` 解除手机端 `RESOLVING` 状态锁。

## 🎯 第四部分：高可用工程断言与设计自洽性证明

1. **防作弊自洽**：玩家无法通过修改前端代码“刷出”装备。所有的后验物品主张必须经过 Python Engine 的 `professions_matrix` 硬编码标签匹配。不符合职业逻辑的物品（如医生掏出冲锋枪）在规则层就会被直接斩断，无法污染全局状态树。
    
2. **防剧透自洽**：当玩家点击“展示手里的私密线索”时，Engine 拦截该意图，将线索大图转化为 `scene_transition` 分发给 Host（全桌可见大图），但线索底层的文字解析只单播推给特定玩家或 `audience: 'party'` 作用域，确保全桌演出的戏剧悬念。
    
3. **AI 容错自洽**：大模型在意图路由层（Layer 1）和机制编译层（Layer 2）均只作为“无状态的翻译器”使用，其所有的发散输出都会被 Python 算法强行对齐为标准的技能名和物品 ID。哪怕大语言模型产生了短暂的格式幻觉，也只能影响叙事文本的可读性，系统的数据链和数值底层绝对不会发生崩塌。
