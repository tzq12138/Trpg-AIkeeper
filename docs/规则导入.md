### 🏛️ 核心架构：双层规则引擎 (Two-Tier Rule Architecture)

我们将规则分为两类：“世界运作的基础物理法则”和“特定场景的触发机制”。

#### 第一层：底层核心系统规则（Python 硬编码插件）

**什么是核心规则？** COC 7版的“D100 检定”、“困难/极难成功划分”、“大成功/大失败区间”、“扣 SAN 机制”、“孤注一掷”等。这些是不可变的基础法则。

**怎么写？** 在 Python Engine 中，采用**策略模式（Strategy Pattern）**与**注册表（Registry）**。我们不写又臭又长的 `if/else`，而是将每个核心机制写成一个独立的 `Handler`。

Python

```
# mc_system/engine/rules/handlers.py

class BaseRuleHandler:
    async def execute(self, state: GameState, params: dict) -> RuleResult:
        raise NotImplementedError

class CocSkillCheckHandler(BaseRuleHandler):
    """COC 7版标准技能检定"""
    async def execute(self, state: GameState, params: dict) -> RuleResult:
        skill_name = params.get("skillName")
        difficulty = params.get("difficulty", "regular")
        
        # 1. 获取玩家面板数值
        skill_value = state.character.skills.get(skill_name, 0)
        
        # 2. 计算门限 (困难=1/2, 极难=1/5)
        target = skill_value
        if difficulty == "hard": target = skill_value // 2
        elif difficulty == "extreme": target = skill_value // 5
            
        # 3. 掷暗骰 (绝对公平的 RNG)
        roll = random.randint(1, 100)
        
        # 4. 判定成功等级 (包含大成功/大失败的硬逻辑)
        if roll == 1:
            level = "critical_success"
        elif roll >= 96 and skill_value < 50:
            level = "fumble" # 技能低于50，96-100大失败
        elif roll == 100:
            level = "fumble"
        elif roll <= target:
            level = "success"
        else:
            level = "failure"
            
        # 5. 返回绝对事实
        return RuleResult(
            is_success=(level in ["success", "critical_success"]),
            metadata={"roll": roll, "target": target, "level": level}
        )

# 引擎启动时注册
rule_registry = {
    "skill_check": CocSkillCheckHandler(),
    "sanity_check": CocSanityCheckHandler(),
    "combat_damage": CocCombatHandler()
}
```

**这一层的好处**：如果未来我们要支持 DND 5e（扔 D20）或者 龙与地下城，只需要写一套新的 `DndSkillCheckHandler` 注册进去即可。

#### 第二层：剧本级业务规则（JSON5 / 受限 DSL 配置）

**什么是剧本级规则？** “玩家阅读这本《死灵之书》残页，触发一次 SAN 值检定，成功扣 1d3，失败扣 1d10，并获得 Cthulhu 神话技能 +5”。

**怎么写？** **绝对禁止创作者在后台写 Python 脚本**（防止沙箱逃逸和安全漏洞）。创作者打包剧本资产时，规则必须以结构化的 JSON5 或极简 DSL 定义在 `scenario_assets.json` 中。

代码段

```
// 剧本节点配置示例
{
  "nodeId": "scene_library_book",
  "triggers": [
    {
      "condition": { "$action": "read_item", "itemId": "book_necronomicon" },
      "mechanics": [
        {
          "type": "sanity_check", // 路由到 CocSanityCheckHandler
          "params": { "successLoss": "1d3", "failureLoss": "1d10" }
        },
        {
          "type": "apply_patch",  // 直接应用属性补丁
          "params": { "path": "/skills/cthulhu_mythos", "op": "add", "value": 5 }
        }
      ]
    }
  ]
}
```

当玩家执行动作时，Engine 先遍历当前 Scene 的 `triggers`。一旦命中，直接按序调用 `rule_registry` 中的 Handler 结算。

### 🔄 规则引擎的数据流闭环 (The Execution Pipeline)

有了这套架构，当大模型（LLM）“机制编译器”输出 `{"triggeredMechanic": "skill_check", "skillName": "射击", "difficulty": "hard"}` 时，规则引擎的工作流如下：

1. **路由分发 (Dispatch)**： 引擎通过 `rule_registry["skill_check"]` 找到对应的 Python 类。
    
2. **状态注入 (Injection)**： 将玩家的 `CharacterNode`（血量/技能表） 作为只读状态注入 Handler。
    
3. **计算事实 (Compute Fact)**： Handler 掷出 `78`，判定为 `failure`。
    
4. **编译突变 (Compile Mutation)**： Handler 将结果编译为两部分输出：
    
    - **视觉事务 (Host 演出)**：生成 `RevealStep`（`kind: 'roll', payload: { rolledValue: 78, resultType: 'failure' }`）。
        
    - **状态突变 (Player 补丁)**：如果有扣血/扣 SAN，生成标准的 RFC 6902 JSON Patch（`[{op: "replace", path: "/hp/current", value: 8}]`）。
        
5. **合流给大模型 (Merge to LLM)**： 引擎将 `[动作: 射击, 难度: 困难, 结果: 78/50 失败]` 送给大模型，让大模型生成枪卡壳或打偏的恐怖描述。
    

### 🛡️ 为什么这么设计？（防坑指南）

1. **绝对确定性**：所有的加减乘除、成功失败，都在 Python 的强类型内存里完成。AI 的幻觉永远无法把 1+1 算成 3，也永远无法在判定失败时给玩家发奖励。
    
2. **沙箱绝对安全**：创作者配置的剧本规则只是 JSON 描述符，Python 只是去解析它。黑客无法通过上传包含恶意 `eval()` 或 `exec()` 的剧本包来控制你的服务器。
    
3. **彻底解耦**：大模型不需要知道什么是“大失败”，不需要去掷随机数。它只负责阅读 Python 甩给它的结算清单，然后“看图说话”。
    

通过这种 **“底层 Python 强控 + 顶层 JSON 编排”** 的模式，规则的写入将变得极其清晰、安全且易于扩展。
